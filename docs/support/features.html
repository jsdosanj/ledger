Every feature, step by step

A complete tour of everything Ledger Governance does inside Obsidian, in plain English with numbered steps. Use the jump list below, or read straight through. Everything here runs 100% locally — no account, no token, no network.

On this page: Governed-document template & frontmatter · Status lifecycle · Ownership · Review tracking & stale detection · Dashboard, ribbon & status bar · The audit log · The 100% local guarantee

The governed-document template & frontmatter

Governance lives in standard YAML frontmatter, so your notes stay portable plain Markdown — readable in any editor, with or without the plugin. The New governed document command scaffolds a note with that frontmatter already in place.

  1. Open the Command Palette (Ctrl/Cmd+P) and run Ledger: New governed document.
  2. Give it a title; the plugin creates the note with governance fields pre-filled.
  3. Edit the body as a normal note — the frontmatter is just data at the top.

The fields it uses:

FieldMeaning
titleThe document's title.
ownerWho is responsible for the note — a name or email. See Ownership.
statusWhere the note sits in its lifecycle: draft, in-review, approved, or published. See Status lifecycle.
typeThe kind of document (for example a policy, runbook, or reference).
reviewed_onThe date the note was last reviewed. See Review tracking.
reviewerWho performed the review.
review_cadence_daysOptional per-note review interval; overrides the default stale threshold for just this note.

Because these are ordinary frontmatter keys, you can also add them by hand to existing notes — the plugin will pick them up.

The status lifecycle

Each governed note moves through four lifecycle stages, recorded in its status field:

draft → in-review → approved → published

  1. A new governed document starts as draft.
  2. When it's ready for eyes, set it to in-review.
  3. Once it's been reviewed and accepted, set it to approved.
  4. When it's the live, authoritative version, set it to published.

To change status: open the note, run Ledger: Set document status, and pick the stage. The change is written to frontmatter and recorded in the audit log. The dashboard's lifecycle mix shows how many notes sit in each stage (including any with no status set).

Ownership

Every governed note should have an owner — the person accountable for keeping it correct. Ownership answers an auditor's first question: who is responsible for this document?

  1. Open the note and run Ledger: Set document owner.
  2. Enter a name or an email address.
  3. The plugin writes it to the owner frontmatter field and records the change in the audit log.

If you turn on Require an owner in settings, any note without an owner is counted as ungoverned in the dashboard and status bar — a simple way to find the gaps.

Review tracking & stale detection

Documents drift out of date. Ledger Governance tracks when each note was last reviewed and flags the ones that have gone stale.

  1. Open a note and run Ledger: Mark current note reviewed.
  2. The plugin stamps today's date into reviewed_on (creating the frontmatter block if needed) and records it in the audit log.
  3. The note's review clock resets — it counts as fresh again.

From reviewed_on, the plugin classifies each note's review freshness:

FreshnessMeaning
freshReviewed within the threshold.
staleHas a reviewed_on date, but it's older than the threshold.
neverNo valid reviewed_on date is set.

The threshold defaults to 90 days and is set in Settings → Community plugins → Ledger Governance → Review-stale threshold. A single note can override the default by setting its own review_cadence_days — for example a fast-moving runbook might use review_cadence_days: 30 while everything else stays at 90.

The governance dashboard, ribbon & live status bar

The Governance dashboard gives you the whole vault's health at a glance. Open it from the Command Palette (Ledger: Governance dashboard) or by clicking the ribbon icon (the book in the left ribbon).

  • Totals — how many notes the plugin sees.
  • Ungoverned — notes missing a required owner or status.
  • Review freshness — counts of never / stale / fresh notes.
  • Lifecycle mix — counts by status across draft, in-review, approved, published, and unset.

Click any row in the dashboard to open that note and act on it.

The status bar at the bottom of the Obsidian window keeps a live one-line summary as you work — for example Ledger: N docs · X stale · Y ungoverned — so you always see where things stand without opening the dashboard.

The in-vault audit log

Every governance action — creating a governed document, setting a status or owner, marking a note reviewed — appends a row to _ledger/audit.md in your vault. It's an ordinary Markdown table with columns date · action · document · detail.

  1. Run Ledger: Open governance audit log to read it.
  2. Each row is one action, newest appended at the end.
  3. Because it's plain Markdown in your vault, you can search, version, or archive it like any note.

The audit log never leaves your vault — it's yours. See the plugin guide for an example table.

The 100% local guarantee

Ledger Governance does everything above on your own machine. There is no account, no token, and no network call — nothing is uploaded and nothing phones home. Your governance metadata lives in your notes' frontmatter, the audit log lives in your vault, and settings are stored in local plugin data. The plugin is free and open source under the MIT License.
